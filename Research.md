# 需求研究（Research） — SqueezePic 图片压缩工具

## 1. 用户画像
- **主要用户**：前端开发者、设计师、内容创作者、网站管理员、日常需要处理大量图片的用户。
- **访问动机**：
  - 减小图片体积，加速网页加载，节省存储空间和带宽。
  - 批量处理多张图片，提升工作效率。
  - 格式转换以适配不同平台需求（如 PNG 透明转 JPG 背景）。
  - 简单编辑（裁剪、旋转）后直接压缩下载，无需打开专业软件。
- **行为特征**：
  - 快速上传 → 调整参数 → 预览 → 下载，期望流程流畅无阻。
  - 对隐私敏感，不希望图片上传到服务器。
  - 移动端也有临时处理图片的需求。

## 2. 竞品分析
- **TinyPNG / TinyJPG**：在线单张/批量压缩，智能算法，质量好但无格式转换和编辑，有上传隐私顾虑。
- **Squoosh (Google)**：开源，纯客户端，支持格式转换、质量滑块、高级编解码对比，但界面稍专业，无批量处理。
- **iLoveIMG / ILovePDF 系列**：批量处理、多种工具集合，但依赖后端，有文件上传。
- **本地软件 (ImageOptim, RIOT)**：强大但需安装，不能即用。

**差异化机会**：
- 纯浏览器端处理，零隐私风险。
- 支持批量上传 + 单独或统一设置压缩质量。
- 集成格式转换、简单裁剪和旋转。
- 现代简洁界面，实时预览对比，一键下载所有文件或打包。

## 3. 功能优先级
- **P0 核心**：本地上传图片、调整压缩质量（滑块）、预览压缩前后效果、单张下载。
- **P1 重要**：批量上传和管理（多文件列表）、批量下载（单个或打包 ZIP）、格式转换（PNG ↔ JPG ↔ WebP）。
- **P2 增值**：图片裁剪（自由裁剪区域）、旋转（90度增量/自由旋转）、重置原图。
- **P3 优化**：拖拽上传、粘贴图片、文件大小/数量限制提示、响应式适配、暗色主题。

## 4. 技术可行性
- 全部基于浏览器 Canvas + Web API 实现，无需后端。
- Canvas 绘制图片后进行 `toBlob()` 输出不同格式和质量。
- 裁剪和旋转可借助 Canvas 变换。
- 批量打包使用 JSZip 库。
- 文件拖拽使用 Drag and Drop API，剪贴板粘贴使用 Clipboard API。
- 纯静态站点，可部署至 GitHub Pages 或 Vercel。

## 5. 浏览器原生编码器局限性与专业编码器选型

### 5.1 问题发现（v1.0 实测）
v1.0 完全依赖浏览器 Canvas `toBlob()`，暴露三大问题：
- **JPEG 体积不稳定**：同一张图、同 quality=80、format='original' 跑两次，结果大小可能差 5%-15%——浏览器对 `image/jpeg` 的编码器未指定算法（Chromium 用 libjpeg-turbo，Firefox 行为不同），无法保证可预测。
- **"保持原格式"反而变大**：当原图是已经高度压缩的 JPEG/PNG，再次走 Canvas 编码，等于重新走一遍有损量化，输出体积经常 > 原图。这违反"压缩"的直觉。
- **PNG quality 无效**：Canvas 在 `image/png` 时忽略 quality 参数，等于无脑重编码，无优化空间。

### 5.2 选型评估

| 方案 | 浏览器可用 | 体积稳定 | 体积上限 | 备注 |
| ---- | ---- | ---- | ---- | ---- |
| Canvas `toBlob` | ✅ | ❌ 波动大 | 仅浏览器自带 | 默认方案 |
| @squoosh/lib (Google) | ❌ | ✅ | ✅ 极致 | 依赖 Node `worker_threads`、`os.cpus()`，**浏览器不可用** |
| **@jsquash/jpeg/png/webp** | ✅ | ✅ | ✅ 极致 | WASM 编译到浏览器，Google 官方维护的 Squoosh 子集 |

### 5.3 选型结论：@jsquash/*（WASM）
- **作者**：Google Chrome 团队（Postlight），与 Squoosh 网页版同源
- **覆盖**：MozJPEG（JPEG）、OxiPNG（PNG 无损）、libwebp（WebP 有损）
- **API**：直接接受浏览器原生 `ImageData`，输出 `ArrayBuffer`，可与 Canvas `getImageData` 无缝衔接
- **体积**：三个 WASM 总 ~700KB，gzip 后 ~250KB，按需懒加载不阻塞首屏
- **隐私**：纯客户端，不联网

### 5.4 替代方案与权衡
- 改用 `OffscreenCanvas + Worker`：性能好但代码量翻倍，对压缩率无改善，否决
- 上 WASM 自己编译 MozJPEG：可控但维护成本高，@jsquash 已能覆盖 95% 场景，否决
- 强制把"输出>原图"时回退到原文件：会无声吞掉用户编辑（旋转/裁剪），否决

### 5.5 验收标准（v1.1）
- JPEG @ quality=80：输出 ≤ 原图 60%-90%
- PNG（保持原格式）：输出 ≤ 原图（OxiPNG 无损）
- WebP @ quality=80：输出 ≤ 原图 30%-60%
- 多次同输入 → 同输出（结果可复现）
